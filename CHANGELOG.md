# Changelog

## v1.6.7
- <strong>Quieter "15 s to delete" hint for privileged users</strong> — The post-note toast suffix "— 15s to delete before notifications send" now only appears for drive-by editors. Admins, supervisors, the WO originator, and the assigned salesperson just see "Note posted" / "Reply posted" without the timing coaching — they already know the drill and (in the case of admins + supervisors) keep unlimited delete authority past the window anyway. Server returns a <code>privileged_author</code> flag from <code>POST /api/work-orders/&lt;id&gt;/general-notes</code> so the frontend doesn't have to re-derive the role relationships.

## v1.6.6
- <strong>Not-deliverable is now a pending state, not an instant archive</strong> — Marking a WO not deliverable no longer files it away immediately. It keeps the WO in the active list with a pale-red tint, a "Not Delivered" badge, and starts collapsed; the existing 23:00 auto-archive sweep rolls it into the archive along with the day's delivered WOs. New column <code>was_not_deliverable</code> (migration v34) latches when the flag is set so the badge + tint survive into the archive view.
- <strong>Pending WOs sort to the bottom of the active list</strong> — Both delivered-pending and not-deliverable-pending cards sort below everything else regardless of the user's chosen sort, so the top of the list is always live work.
- <strong>15-second retraction window on notes + replies + flag notes</strong> — Posting a note no longer fires the email and in-app notifications synchronously. The author has 15 seconds to delete the note; if they do, the email and fan-out notifications are suppressed entirely. If not, a server-side Timer delivers everything at the 15-second mark. The Delete button on freshly-posted notes auto-greys when the window closes (1 Hz sweep keyed off a <code>data-expires-at</code> attribute). Admins keep their unlimited delete. Backend enforces the window too — non-admin authors posting a DELETE past the deadline get a 403.
- <strong>Admin-only "notifications globally off" banner</strong> — The yellow banner that warns "Email notifications are off globally" now only renders for admins. Editors and viewers don't see it (it's a setting they can't change). Likewise, "email not sent" suffixes are stripped from status toasts when the global flag is known to be off, so editors aren't nagged about suppressed emails they didn't control.
- <strong>Cleaner audit trail for deleted notes</strong> — The note-delete confirmation dialog no longer mentions the audit log (the copy was implementation-leaky). Behind the scenes every deletion is still recorded: deleting a note appends a <code>note_deleted</code> audit row and leaves the original <code>note_added</code> row in place, so the log reads as a full add→delete timeline instead of scrubbing evidence.

## v1.6.5
- <strong>Live in-app notification cascade for work-order updates</strong> — Toast-style notifications slide in from the top-right whenever a work order you're watching changes. Recipients per event are: the originator, the assigned salesperson, and every user who has ever authored a note or reply on that WO's thread — so once you engage, you stay in the loop. Triggers cover every meaningful event: new note, reply, flag note, photo upload, part flagged / unflagged, marked delivered, marked not deliverable, reopened, and body edits. Clicking the toast jumps straight to <code>/workorders/&lt;wo_number&gt;</code>; the ✕ or the 15-second auto-dismiss timer marks the notification read so it doesn't pop again on the next poll. Each card is color-coded by kind (green for delivered, red for flag / not-deliverable, amber for reopened, blue for notes / replies / photos / edits) with a kind icon, actor name, and relative timestamp. Polls every 5 seconds (matching the WO live-pulse cadence) so updates feel near-real-time, and the initial page-load batch is capped at 5 so logging back in after a busy day doesn't bury the screen.
- <strong>Unread-notification badge on the Work Orders sidebar link</strong> — A red pill next to "Work Orders" shows the current unread count for the signed-in user. Clicking a toast, pressing ✕, or letting the 15-second auto-dismiss run decrements the badge immediately (optimistic UI); the next 5-second poll reconciles against the server's canonical count so multi-tab usage and background activity stay in sync.
- <strong>Backing API</strong> — New <code>notifications</code> table (migration v33; row per recipient per event, <code>read_at</code> tracks acknowledgement). Endpoints: <code>GET /api/notifications/unread?since=&lt;id&gt;</code> returns new items + <code>max_id</code> + <code>unread_total</code>; <code>POST /api/notifications/&lt;id&gt;/read</code> marks one read; <code>POST /api/notifications/read-all</code> is seeded for a future inbox.

## v1.6.4
- <strong>First-run setup wizard</strong> — Fresh installs now walk admins through a 5-step onboarding on first login: replace the default <code>admin/admin</code> password (mandatory; enforces 12+ chars, 3-of-4 character classes, blocks "admin" and a handful of other clichés), set a display name and real email for the admin account, configure the public URL used by email deep-links, wire up SMTP, and pick a timezone + 12/24h clock. Steps 2–5 are all skippable and pre-filled from whatever the install already has. Progress dots, per-step auto-focus, animated card entry, live password-strength meter, and a finish screen that lands on the dashboard. Driven by a new <code>setup_complete</code> app setting: a migration seeds existing installs as <em>complete</em> (so upgrades don't shove live users back into onboarding) and only fresh installs — still running the default admin password — trigger the wizard. Admin-only; non-admins hit a neutral "still being set up" screen until the flag flips. All <code>/api/setup/*</code> endpoints 409 once setup is done, so there's no backdoor to re-run the flow after the fact.
- <strong>Assigned salesperson can now edit their own work orders</strong> — Previously only the originator, admins, and supervisors could rewrite a WO's request fields or parts list; plain editors (including the salesperson named on the WO) saw a disabled Edit button. The edit gate now also accepts the salesperson assigned to that specific work order, matched case-insensitively against either their display name or their username. Backend <code>PUT /api/work-orders/&lt;id&gt;</code> and the frontend <code>canEditBody</code> check both honor the new rule; the disabled-button tooltip was updated to mention the salesperson role. Delete / archive / mark-not-deliverable gates were intentionally left alone — those remain originator-or-above.

## v1.6.3
- <strong>Work-order notification emails now include a deep-link, uniform header, and do-not-reply notice</strong> — Every work-order email (flag / note / reply / delivered / not-deliverable / photo / on-demand update) now opens with the same header block: WO #, Quote/Invoice #, Customer. Each email ends with "— Do not reply to this email. Log in to Warehouse Manager to respond." followed by a direct link to the work order. The link's base URL comes from a new <strong>Public URL</strong> field in <em>Settings → Admin → SMTP / Email</em>; when left blank it falls back to the host used when the email was triggered, so upgrades don't break anything mid-flight.
- <strong>Single work-order view at <code>/workorders/&lt;wo_number&gt;</code></strong> — The email deep-links land on a new route that reuses the exact same card renderer as the list view, but shows only the linked work order (expanded by default, with a "← All Work Orders" link back to the list). URLs use the human-readable WO number (e.g. <code>/workorders/WO-00042</code>) so shared links stay meaningful; legacy numeric-id URLs still resolve and transparently rewrite to the canonical form. Auth is still enforced by Flask-Login, so an unauthenticated click bounces through <code>/login</code> and back. The ellipses (⋯) slideout on every WO card gets a matching <strong>View Work Order</strong> action, so users can grab the shareable URL without hunting for it; the button hides when you're already on the single-WO page.
- <strong>Click-to-toggle WO cards</strong> — In both the active and archive lists: clicking anywhere on a collapsed card expands it; clicking the title bar of an expanded card collapses it. The chevron button keeps working either way. The body of an expanded card (notes thread, parts list, attachments) stays non-toggling so scrolling through content doesn't accidentally close the card. The chevron itself now mirrors state truthfully — down (▼) when collapsed, up (▲) when expanded.
- <strong>WO # on each card is now a link to its own page</strong> — The WO number chip used to copy itself to the clipboard; it now navigates to <code>/workorders/&lt;wo_number&gt;</code> instead (matching the email deep-link target). Renders as a plain span on the single-WO page so it doesn't link to itself. Text alignment of the WO # chip was also tightened so its text lines up with the vehicle / meta row directly below instead of sitting 7 px further in from the chip's padding.

## v1.6.2
- <strong>Modals and overlays no longer stutter on integrated graphics</strong> — Removed <code>backdrop-filter: blur()</code> from the modal overlay, the per-modal "busy" spinner overlay, and the bulk-progress overlay. Full-viewport backdrop blur forces the compositor to read back and re-blur the framebuffer every frame, which Intel/iGPU shared-memory setups handle badly (Safari on Apple Silicon and discrete GPUs have a fast path, so the issue didn't show up there). Overlays are now solid translucent fills (<code>rgba(0,0,0,0.72)</code> for modals, <code>92%</code> card-colour for the in-modal busy layer) — visually very similar, but modals open smoothly on shop-floor hardware.

Wheel, touch, and keyboard scrolling still work; the 70 vh height cap is unchanged.

- <strong>In-session update watcher</strong> — A tiny <code>GET /api/version</code> endpoint plus a once-a-minute frontend check detects when the deployed bundle has moved past the version a user's tab booted with. A non-blocking floating banner ("Update available — vX.Y.Z · Reload when you're at a stopping point") appears at bottom-right with <strong>Later</strong> and <strong>Reload now</strong> buttons. Never auto-reloads — users stay in control so mid-typed notes and open modals are never pulled out from under them. Dismissing re-prompts after 10 minutes.

## v1.5.17
- <strong>Live-refresh on Work Orders</strong> — The Work Orders page now auto-refreshes when anyone else adds, edits, deletes, flags, archives, or notes on a work order — no manual reload. A lightweight signature endpoint <code>GET /api/work-orders/pulse</code> (returns <code>count</code> + <code>MAX(updated_at)</code>) is polled every 5 seconds; the full list only re-fetches when the signature changes. Polling pauses while any modal is open (so edits aren't yanked out mid-typing) and while the browser tab is hidden, and resumes immediately when the tab regains focus. Per-card expand/collapse state and the open pill slideout are preserved across refreshes, and the sidebar status counts refresh alongside the list.
